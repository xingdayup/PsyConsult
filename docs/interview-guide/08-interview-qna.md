# 08. 面试 Q&A 速查

## 1. 项目名称及一句话介绍

**答法：**  
项目叫 PsyConsult，是一个面向精神科初诊和用药审查场景的临床决策支持系统。医生输入患者症状、病程或治疗方案后，系统通过 LangGraph 多 Agent 工作流，结合 RAG、知识图谱和会话记忆，流式输出症状抽取、ICD-11 鉴别诊断、治疗建议和药物相互作用审查。

## 2. 你在项目中的角色

**答法：**  
这个项目主要由我独立完成。我负责需求拆解、系统架构、Agent 工作流设计、Prompt 设计、FastAPI 后端、Vue 前端、Redis/Milvus/Neo4j 数据接入、SSE 流式输出和 Cloudflare 部署排障。

如果需要更谦虚：  
项目开发过程中我也使用 AI 工具辅助编码和排障，但核心架构设计、模块拆分、Prompt 约束、部署调试和代码整合由我负责。

## 4. 这个项目解决了什么痛点？

**答法：**  
精神科问诊中，医生面对的是非结构化描述，需要快速完成症状标准化、量表推断、鉴别诊断、治疗建议和药物风险判断。普通 LLM 直接回答不够透明，也容易幻觉。这个项目把临床任务拆成多个 Agent，用工具和 Prompt 约束每一步，让系统按临床流程输出可追踪的辅助建议。

## 5. 你如何验证用户需要？

**答法：**  
我从精神科初诊流程拆解需求，选取抑郁、双相鉴别、焦虑、精神病性症状、强迫症状和药物联用等典型场景做测试。联调和测试时也发现用户不只需要最终答案，还需要知道系统正在做什么，所以加入了流式阶段提示和后端耗时日志。

## 6. 为什么用多 Agent？

**答法：**  
因为临床辅助决策不是单一任务。诊断、治疗和药物审查关注点不同，所需工具也不同。多 Agent 可以让每个 Prompt 更短、更明确，也能让输出结构和工具调用更可控。

## 7. 为什么用 LangGraph？

**答法：**  
LangGraph 适合表达有状态、多节点、条件路由的 Agent 工作流。项目用 `AgentState` 传递消息、用户、会话、记忆上下文和路由状态，用 Orchestrator 决定进入哪个临床节点。

## 8. 如何实现流式输出？

**答法：**  
后端用 FastAPI `StreamingResponse` 返回 `text/event-stream`。`stream_chat()` 是异步生成器，每个阶段通过 `yield data: {...}` 推送。Agent 内部用 `get_stream_writer()` 把节点开始、工具调用和模型 chunk 传回 LangGraph custom stream，前端用 `fetch()` 的 `ReadableStream` 逐行解析。

## 9. 为什么不用 WebSocket？

**答法：**  
这个场景是一次请求持续返回文本，SSE 更简单，也更符合 HTTP 部署和代理环境。WebSocket 更适合双向实时通信，而这里主要是服务端持续推送。

## 10. 如何降低幻觉？

**答法：**  
主要从三方面做：第一，Prompt 要求症状必须引用原文 evidence；第二，治疗药物和指南依据必须来自工具检索；第三，图谱或文档查不到时明确标注不确定，不允许编造。

## 11. Redis、Milvus、Neo4j 各自做什么？

**答法：**  
Redis 存短期对话记忆，按用户和会话隔离并设置 TTL。Milvus 用于语义缓存、长期偏好和指南文档向量检索。Neo4j 用于结构化知识图谱，例如疾病-症状、治疗-药物、药物-相互作用关系。

补充讲法：  
长期记忆链路是：当前问答先写入 Redis；后台从 Redis 会话抽取稳定的临床关键信息；再向量化写入 Milvus `long_term_memory`；后续新请求按 `user_id` 过滤并用语义相似度召回相关历史背景。

## 12. 语义缓存和 RAG 的区别？

**答法：**  
语义缓存是相似问题直接复用已有答案，主要优化性能和延迟。RAG 是检索相关知识材料，再让 Agent 结合材料生成答案，主要增强知识依据。

## 13. 前端有什么工程亮点？

**答法：**  
前端实现了 POST SSE 流式解析、多会话本地消息缓存、阶段状态与正文分离展示、Markdown 安全渲染，以及生产 API 地址兜底配置。尤其是多会话切换时，流式返回会写回请求所属 session，不会污染当前会话。

## 14. 安全上做了什么？

**答法：**  
后端把 `user_id` 从 body 移到受控请求头 `X-User-Id`，并做格式校验。可选 `Authorization: Bearer token`。前端对 Markdown 输出使用 DOMPurify 消毒，防止 XSS。CORS 只允许指定前端域名。

## 15. 项目现在的不足？

**答法：**  
它还是原型系统，不能替代医生诊断和处方。当前没有完整登录系统、没有真实病例评测集、临床输出还可以进一步结构化。部署上如果使用临时 Cloudflare Tunnel，API 地址不稳定，长期应该改成固定域名或云上后端。

## 16. 如果面试官问可扩展方向，你会怎么答？

**答法：**  
我会优先做四件事：第一，固定后端 API 域名和服务守护；第二，建设病例评测集评估诊断和推荐质量；第三，加入更严格的临床安全审查和免责声明；第四，把输出改成结构化 JSON 加可视化解释，方便医生复核。
